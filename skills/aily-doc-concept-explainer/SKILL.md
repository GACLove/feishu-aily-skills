# SKILL: aily-doc-concept-explainer

概念解释：由浅入深讲清概念的定义、原理与本质，面向不同知识背景的读者。适用于：用户希望彻底理解某个概念/术语/技术/理论，从直觉层到形式化层逐步深入，同时配合类比、图示、案例等多维度辅助理解。

---

## 1) Overview

### ⚠️ 输出形式（强制）
- **本 skill 被触发即代表用户期望生成飞书云文档**，主 Agent 无需询问用户是否需要写成文档，也不得以对话文本替代文档作为最终交付物。
- 数据收集完成后，必须通过 `task`（subagent_type: writer）委托 Writer Agent 创建飞书云文档。

### 飞书工具优先（必须遵守）
- 所有飞书相关信息获取与写作，先参考 `feishu-use-skills-map` 并按其指引调用对应技能
- 需要访问飞书文档/群消息/会议/任务时，必须先 `get_skills` 获取对应飞书技能说明

### 工具/Skill 明确清单（执行前先 get_skills）
- 必须先参考 `feishu-use-skills-map` 并调用对应技能
- 主要技能及用途：

| 工具 | 用途 | 适用场景 |
|------|------|---------|
| `knowledge_answer` | 检索内部知识库 | 公司内部对该概念的定义、使用惯例、内部文档 |
| `web_search` | 广泛检索公开信息 | 权威定义、原理解释、类比素材、历史演进、应用案例、常见误区 |
| `fetch` | 获取特定网页/文档全文 | 深入阅读权威教程、百科页面、技术博客、学术资料 |
| `aily-diagram` | 生成流程图/结构图/关系图 | 概念关系图谱、工作流程图、层次结构图、对比图 |
| `file` | 将素材写入 draft.md（append 模式） | 每个数据步骤完成后 |

### 证据与时间约束（与主/写作agent一致）
- 任何数据/结论必须来自可追溯来源；不确定则明确标注"需补充资料"。
- 若用户指定时间范围，必须严格筛选在时间窗内的资料，过期内容不得使用。
- 不可编造事实、链接或来源；引用必须有明确出处。
- 定义与原理部分必须引用权威来源（教科书、官方文档、权威百科、学术论文），不得仅依赖二手博客。

### 可读性规则（强制）
- 并列项>=3：优先使用无序/有序列表或表格
- 单段>150字必须拆分（列表/小节/Callout）
- 连续两段无列表且信息密度高：插入列表或 Callout
- 类比与图示穿插在正文中，不集中堆砌
- 公式/模型附带自然语言解释，不裸写公式

### 触发方式（Triggers）
当用户出现以下意图时触发本 skill：
- "由浅入深解释"
- "彻底搞懂"
- "本质是什么"
- "帮我理解XX"
- "XX到底是什么"
- "用通俗的话解释XX"
- "XX的原理是什么"
- "深入讲讲XX"

### 输入（Inputs）
- **必选**：概念/术语（一个明确的概念名称或短语）
- **可选**：
  - 期望深度（入门/标准/深度）
  - 背景知识（读者已有的知识水平，例如"我是非技术人员""我有计算机本科背景"）
  - 困惑点（读者具体不理解的方面，例如"不明白为什么需要XX""搞不清XX和YY的区别"）
  - 应用场景偏好（希望重点关注哪个领域的应用）

### 核心输出（Deliverables）
- **飞书云文档概念解析**：层次递进、结构清晰、多维解释、可读性强

### 质量标准
- **层次递进**：从直觉层（一句话+类比）到机制层（原理+流程）到形式化层（公式/模型）到权衡层（局限+对比），逐层深入，读者可按需停止阅读
- **多维解释**：同一概念至少使用 3 种解释维度——类比（生活化比喻）、图示（结构图/流程图）、形式化（公式/模型/定义），三者互补
- **准确性**：核心定义与原理必须有权威来源支撑，不得凭主观理解编写
- **区分度**：必须明确该概念与易混淆概念的边界

## 3) Module A — 资料获取（主 Agent 执行）

> **核心原则**：优先使用飞书工具获取内部知识，再用外部检索补充权威公开信息。每一步检索完成后，将结构化结果写入 `draft.md`，供后续 Writer Agent 使用。

### A1. 权威定义

**目标**：获取概念最权威、最准确的定义，确保后续所有解释建立在正确的基础上。

**web_search 查询**（中英文双语检索）：
```
- "{概念} 定义 含义"
- "{概念} 是什么 解释"
- "{concept} definition"
- "{concept} meaning explanation"
```

**knowledge_answer 查询**（内部知识库）：
```
knowledge_answer(
  query_list: [
    "{概念} 的定义和含义",
    "公司内部如何定义 {概念}",
    "{概念} 相关的内部文档和规范"
  ],
  explanation: "检索内部知识库中对该概念的定义、使用惯例和相关文档"
)
```

**来源优先级**：
1. 权威教科书/学术论文中的定义
2. 官方文档/标准组织的定义（如 IEEE、W3C、RFC 等）
3. 权威百科（维基百科、百度百科的引用来源）
4. 内部知识库中的定义

## 4) Module B — 结构化分析（主 Agent 执行）

> **核心原则**：将 Module A 收集的素材进行结构化整理，形成由浅入深的讲解层次，确保最终文档逻辑清晰、层次分明。

### B1. 确定讲解层次

根据概念的复杂度和用户的背景知识，确定四层递进结构：

```
┌─────────────────────────────────────────────────────────────┐
│ 第四层：权衡层（Trade-off Layer）                              │
│ - 局限性与边界条件                                             │
│ - 与替代方案的对比权衡                                         │
│ - 适用 vs 不适用场景                                           │
├─────────────────────────────────────────────────────────────┤
│ 第三层：形式化层（Formal Layer）                               │
│ - 数学公式/模型定义                                            │
│ - 严格的技术规格                                               │
│ - 证明/推导（如适用）                                          │
├─────────────────────────────────────────────────────────────┤
│ 第二层：机制层（Mechanism Layer）                               │
│ - 核心原理与工作流程                                           │
│ - 关键步骤与因果关系                                           │
│ - 组成部分与相互作用                                           │
├─────────────────────────────────────────────────────────────┤
│ 第一层：直觉层（Intuition Layer）                              │
│ - 一句话定义                                                   │
│ - 生活类比                                                     │
│ - 图示说明                                                     │
└─────────────────────────────────────────────────────────────┘
```

**层次深度决策规则**：
- 用户说"入门/通俗"：重点展开第一层和第二层，第三四层简要提及
- 用户说"标准"（默认）：四层均展开，第三层中等深度
- 用户说"深度"：四层均深度展开，第三层包含完整公式推导

### B2. 构建类比体系

从 A3 收集的类比候选中，筛选并构建一套互补的类比体系：

**筛选标准**：
1. **准确性**：类比是否正确映射了概念的核心机制？
2. **覆盖度**：多个类比组合后是否覆盖了概念的主要方面？
3. **局限标注**：每个类比的局限是否已明确？（类比一定有局限，必须说明"这个比喻在哪里会失效"）

**类比体系结构**：
- 主类比（覆盖核心机制的生活化比喻，用于"直觉层"）
- 补充类比（覆盖主类比未触及方面的比喻，用于"机制层"的辅助理解）
- 反面类比（"它不像什么"，用于纠正常见误解）

### B3. 整理关键公式/模型（如适用）

并非所有概念都有公式/模型，但若该概念涉及定量描述，则需整理：

**整理规范**：
- 每个公式必须附带"自然语言翻译"（把公式翻译成一句话）
- 每个符号必须在第一次出现时定义
- 公式之间的逻辑关系必须说明（从哪个公式推导出哪个公式）
- 给出公式的直觉解释（"这个公式在说什么"）

**写入 draft.md**（使用 `file(path: "/home/workspace/draft.md", mode: "append")`）：
```markdown
## B3. 公式/模型整理

### 核心公式
公式1：{LaTeX 或文本表述}
- 自然语言翻译：...
- 符号定义：{符号} = {含义}
- 直觉解释：这个公式告诉我们...

### 公式间关系
公式1 → 公式2：通过{条件/推导}得到...
```

### B4. 梳理与相关概念的关系图谱

**关系类型标注**：
- **上位概念**：{概念}属于哪个更大的范畴？
- **下位概念**：{概念}有哪些子类型/变体？
- **并列概念**：与{概念}处于同一层级的其他概念是什么？
- **前置概念**：理解{概念}之前需要先理解什么？
- **后续概念**：理解了{概念}之后可以进一步学习什么？

## 9) 常见失败模式与修复

| 失败模式 | 原因 | 修复方法 |
|---------|------|---------|
| 定义不准确 | 仅依赖二手博客，未查权威来源 | 必须引用教科书/官方文档/标准组织的定义，至少 2 个权威来源交叉验证 |
| 层次不递进 | 所有内容平铺在同一深度 | 严格按直觉层→机制层→形式化层→权衡层组织，每层的目标读者和表达方式不同 |
| 类比误导 | 类比不准确或未标注局限 | 每个类比必须经过"准确性检查"，并明确写出"这个类比在哪里会失效" |
| 公式裸写 | 只写公式不解释 | 每个公式必须附带自然语言翻译、符号定义和直觉解释 |
| 概念边界模糊 | 未与易混淆概念做对比 | 必须至少做 1 组易混淆概念对比表，明确"是什么"和"不是什么" |
| 只有理论无案例 | 缺少应用场景和实际案例 | A5 步骤必须执行，至少 2 个典型场景 + 1 个实际案例 |
| 深度不足 | 只停留在表面定义 | 检查是否完成了"原理与机制"和"深层原理"部分，确保回答了"为什么"而不只是"是什么" |
| 来源缺失 | 定义和数据未标注出处 | 强制格式：每个定义/数据后必须跟（来源：{名称}），无来源的内容标注"需补充资料" |
| 内部视角缺失 | 未查 knowledge_answer | A1 步骤中必须执行 knowledge_answer 检索，补充内部视角 |
| 图示缺失 | 未使用 aily-diagram | A3 步骤必须规划至少 1 张图示，Writer Agent 必须插入 |
