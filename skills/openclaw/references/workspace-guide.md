# 工作区详细指南

## OpenClaw 设计哲学

OpenClaw 的核心洞察：与其把指令硬编码到系统提示中，不如**把 Agent 的身份、行为和记忆外化为用户可编辑的 Markdown 文件**。

两个基础抽象：
1. **外化记忆** — 上下文窗口是缓存，磁盘文件是真相源
2. **声明式身份** — 通过描述性的 Markdown 文件定义 Agent "是谁"，而非程序化指令

## 文件优先级与加载顺序

### 必须加载（每次会话）

| 优先级 | 文件 | 原因 |
|--------|------|------|
| 1 | aily.md | 操作指令和工作记忆，是 aily 行动的基础 |
| 2 | soul.md | 定义交互风格和行为边界 |

### 按需加载

| 文件 | 触发条件 |
|------|----------|
| user.md | 涉及用户偏好、工作习惯时 |
| tools.md | 涉及环境操作、工具使用时 |
| memory/daily/今天.md | 需要回顾今日上下文时 |
| memory/daily/昨天.md | 需要连续性上下文时 |

## 各文件的设计原则

### soul.md 的写作要点

**写"是什么"，不写"做什么"：**
- 好: "我偏好直接、不绕弯子的沟通方式"
- 差: "当用户问问题时，先分析问题类型，然后..."

**保持真实和具体：**
- 好: "我对过度工程化的方案有天然的抵触"
- 差: "我是一个优秀的助手"

**定义边界比定义能力更重要：**
- 什么时候该谨慎
- 什么信息不能外泄
- 什么操作需要确认

### user.md 的演进策略

**从简开始，逐步丰富：**

第一天:
```markdown
## 基本信息
- 张三，前端开发
```

一周后:
```markdown
## 基本信息
- 张三，前端开发 @ 平台团队
- 主要做 React + TypeScript

## 偏好
- 异步沟通，飞书 IM 优先
- 函数式风格，偏好 hooks
- 周五下午不开会
```

一个月后:
```markdown
(完整的用户画像，包含项目、关注领域、个人备注等)
```

### tools.md 的记录原则

**只记"特殊的"，不记"通用的"：**
- 记: "Git 提交需要 GPG 签名" (特殊约束)
- 不记: "使用 git 进行版本控制" (所有人都知道)

**记"别名"和"内部叫法"：**
- "流水线" = Jenkins CI
- "CRM" = 飞书 CRM
- "dev box" = 10.0.1.100 开发机

## 每日日志的最佳实践

### 记什么

- **决策和原因** — "选择方案A因为..." 比 "讨论了方案A和B" 更有价值
- **待办和跟进** — 明确的下一步行动
- **新学到的** — 新术语、新联系人、新流程
- **状态变化** — 项目里程碑、问题解决

### 不记什么

- 流水账（"打开了文件X，修改了Y行"）
- 可以从 git log 重建的信息
- 临时的调试过程

### 格式建议

```markdown
# YYYY-MM-DD

## 讨论主题
- [主题1]: [一句话总结]
- [主题2]: [一句话总结]

## 决策记录
- [决策]: [原因]

## 待办跟进
- [ ] [具体行动] — [截止时间/优先级]

## 新增记忆
- [类型]: [内容]
```

## 上下文压缩策略

### 什么时候压缩

- 对话超过 30 轮
- 开始遗忘早期对话内容
- 用户明确要求
- 话题发生重大转换

### 压缩前必做

1. **扫描新信息** — 对话中提到的新术语、新人名、新项目
2. **提取决策** — 做出的选择和原因
3. **记录待办** — 未完成的任务和承诺
4. **写入持久化** — 上述信息写入对应的 memory/ 文件

### 压缩后保留

- 当前活跃的讨论主题
- 未完成的任务上下文
- 最近的决策和约束条件

## 多项目工作区

如果用户在多个项目间切换，每个项目可以有自己的工作区：

```
~/project-a/
├── aily.md          ← 项目A的操作指令
├── tools.md         ← 项目A的环境配置
└── memory/          ← 项目A的记忆

~/project-b/
├── aily.md          ← 项目B的操作指令
├── tools.md         ← 项目B的环境配置
└── memory/          ← 项目B的记忆

~/.aily/
├── soul.md          ← 全局身份（可选，跨项目共享）
└── user.md          ← 全局用户画像（可选，跨项目共享）
```

**原则**：soul.md 和 user.md 通常跨项目一致，tools.md 和 aily.md 因项目而异。

## 与 OpenClaw 的对照

| OpenClaw | aily 工作区 | 说明 |
|----------|-------------|------|
| SOUL.md | soul.md | 身份与个性 |
| AGENTS.md | aily.md | 操作指令 |
| USER.md | user.md | 用户画像 |
| TOOLS.md | tools.md | 环境备注 |
| IDENTITY.md | (合并到 soul.md) | 简化，不需要单独的外观定义 |
| MEMORY.md | memory/ 目录 | 由 memory-management 技能管理 |
| memory/YYYY-MM-DD.md | memory/daily/YYYY-MM-DD.md | 每日日志 |
| BOOTSTRAP.md | 初始化流程 | 内置在技能中，不需要单独文件 |
| HEARTBEAT.md | (不适用) | aily 没有后台心跳机制 |
| BOOT.md | (不适用) | aily 没有网关重启概念 |
