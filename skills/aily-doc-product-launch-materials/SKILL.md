# SKILL: aily-doc-product-launch-materials

产品发布会材料：聚焦核心卖点提炼、发布会全流程设计、现场演示脚本编排与对外FAQ统一口径，适用于**对外宣讲、线上/线下发布会、内部产品宣讲会**等场景。输出一份可直接用于发布会筹备与执行的结构化飞书云文档。

---

## 1) Overview

### ⚠️ 输出形式（强制）
- **本 skill 被触发即代表用户期望生成飞书云文档**，主 Agent 无需询问用户是否需要写成文档，也不得以对话文本替代文档作为最终交付物。
- 数据收集完成后，必须通过 `task`（subagent_type: writer）委托 Writer Agent 创建飞书云文档。

### 飞书工具优先（必须遵守）
- 所有飞书相关信息获取与写作，先参考 `feishu-use-skills-map` 并按其指引调用对应技能
- 需要访问飞书文档/群消息/会议/任务时，必须先 `get_skills` 获取对应飞书技能说明

### 工具/Skill 明确清单（执行前先 get_skills）
- 必须先参考 `feishu-use-skills-map` 并调用对应技能
- 主要技能：

| 工具 | 用途 | 适用场景 |
|------|------|---------|
| `aily-doc` | 检索飞书云文档 | PRD、技术方案、以往发布会材料、产品文档 |
| `knowledge_answer` | 检索内部知识库 | 产品定位、技术参数、内测数据、客户反馈 |
| `web_search` | 广泛检索公开信息 | 竞品对比、市场趋势、行业案例、用户评价 |
| `fetch` | 获取特定网页全文 | 竞品官网产品页、新闻稿、评测文章、白皮书 |
| `file` | 将素材写入 draft.md（append 模式） | 每个数据步骤完成后 |

### 证据与时间约束（与主/写作agent一致）
- 任何数据/结论必须来自可追溯来源；不确定则明确标注"需补充资料"
- 若用户指定时间范围，必须严格筛选在时间窗内的资料，过期内容不得使用
- 不可编造事实、链接或来源；引用必须有明确出处
- 产品参数必须与最新PRD/技术文档一致，不得使用历史版本数据

### 可读性规则（强制）
- 并列项>=3：优先使用无序/有序列表或表格
- 单段>150字必须拆分（列表/小节/Callout）
- 连续两段无列表且信息密度高：插入列表或 Callout
- 发布会流程、演示脚本等时间敏感内容，必须使用表格呈现

### 触发方式（Triggers）
- "发布会材料"
- "产品发布"
- "发布会文案"
- "产品宣讲"
- "帮我准备一场产品发布会"
- "写一份发布会材料"
- "产品上线发布材料"

### 输入（Inputs）
- **必选**：
  - 产品名称（例如：飞书多维表格 3.0）
  - 发布时间/场景（例如：2025年3月15日线上发布会）
  - 核心卖点（例如：AI自动建表、跨表关联分析、实时协同编辑）
- **可选**：
  - 目标受众描述（例如：中大型企业IT负责人、项目管理者）
  - 竞品对比点（例如：对标Airtable、Notion Database）
  - 发布会时长限制（例如：90分钟）
  - 演示场景偏好（例如：侧重企业项目管理场景）
  - 历史发布会参考（例如：参考去年飞书大会风格）

### 核心输出（Deliverables）
- **飞书云文档发布会材料**：包含完整的发布目标、受众画像、核心卖点提炼、产品演示脚本、发布会流程、客户案例、对外FAQ及统一口径

### 质量标准
- **信息准确**：所有产品卖点有PRD或技术文档支撑，所有数据有来源标注
- **节奏清晰**：发布会流程可直接用于现场执行，时间分配合理
- **卖点锐利**：每个核心卖点有"一句话总结 + 数据支撑 + 用户故事"三要素
- **口径统一**：FAQ覆盖技术、商业、竞品三类高频问题，回答可直接对外使用

## 3) Module A — 资料获取（主 Agent 执行）

> **核心原则**：优先使用飞书工具获取内部产品资料，外部检索作为竞品与市场信息的补充。产品自身信息以内部文档为准，严禁用外部搜索结果替代内部权威数据。

### A1. 产品信息

**目标**：获取产品的完整定义、核心功能、技术参数与定位描述。

#### aily-doc 搜索 PRD/技术方案/产品文档

通过 `get_skills` 获取 `aily-doc` 技能后调用：

```
搜索飞书云文档："{产品名称} PRD 产品需求文档"
搜索飞书云文档："{产品名称} 技术方案 架构设计"
搜索飞书云文档："{产品名称} 产品介绍 功能说明"
过滤条件：doc_type = "docx"，优先选择最近更新的文档
```

**提取重点**：
- 产品一句话定义
- 核心功能列表（功能名称 + 简述）
- 技术架构要点（用于技术类FAQ回答）
- 版本迭代亮点（与上一版本的差异）
- 目标用户画像（产品侧定义的核心受众）

```markdown
**场景二：全员Sprint规划（千人协同）**

**场景三：安全审计场景（数据合规）**
```

## 9) 常见失败模式与修复

| 失败模式 | 具体表现 | 原因分析 | 修复方法 |
|---------|---------|---------|---------|
| 卖点不锐利 | 卖点描述像功能列表，缺乏感染力 | 只堆砌功能参数，缺少用户故事和对比 | 严格执行"一句话+数据+故事"三要素，每个卖点必须有竞品对比 |
| 数据与PRD不一致 | 发布会材料中的参数与实际产品不符 | 使用了过期文档或外部未经验证的数据 | 产品参数必须以最新PRD为准，通过 aily-doc 获取最新版本 |
| FAQ口径不统一 | 不同章节对同一问题的回答有矛盾 | FAQ和正文内容分别编写，未做一致性校验 | 统一在FAQ章节定义口径，正文引用时保持一致 |
| 演示脚本无备选 | 现场Demo出现技术故障无法继续 | 只准备了在线演示，未考虑异常场景 | 每个Demo场景必须准备录屏备选方案和异常处理预案 |
| 流程时间分配失衡 | 技术演示过长导致Q&A被压缩 | 未严格控制各环节时长 | 流程表必须精确到分钟，排练时严格计时 |
| 市场背景空洞 | 只有定性描述，缺乏数据支撑 | web_search 检索不充分，未获取权威报告数据 | 市场数据至少2个来源交叉验证，必须标注来源和年份 |
| 竞品对比不公正 | 过度贬低竞品，容易引发公关风险 | 使用攻击性语言或不实对比 | 竞品对比必须基于公开可验证数据，禁止攻击性用语 |
| 客户案例缺授权 | 使用了未经客户授权的案例信息 | 直接引用内部数据未确认对外授权 | 所有客户案例需确认对外使用授权，敏感信息做脱敏处理 |
| 内部视角缺失 | 未利用内部知识库的产品数据 | 只做了外部搜索，忽略了 knowledge_answer 和 aily-doc | 严格执行A1（产品信息）和A4（案例数据）的内部检索步骤 |
| 受众画像模糊 | 材料没有针对性，对所有人说同样的话 | 未区分不同受众的关注点差异 | 目标受众画像表必须包含每类受众的核心关注点和沟通策略 |
