# SKILL: aily-doc-company-research

公司调研：基于官方与公开信息输出公司概况、业务结构、竞争位置、经营与风险的结构化研究。面向指定公司进行**全维度、多来源**的系统性调研，覆盖基本面、业务产品、竞争格局、经营财务、风险动态等关键维度，最终输出一份可直接交付的飞书云文档公司调研报告。

---

## 1) Overview

### ⚠️ 输出形式（强制）
- **本 skill 被触发即代表用户期望生成飞书云文档**，主 Agent 无需询问用户是否需要写成文档，也不得以对话文本替代文档作为最终交付物。
- 数据收集完成后，必须通过 `task`（subagent_type: writer）委托 Writer Agent 创建飞书云文档。

### 飞书工具优先（必须遵守）
- 所有飞书相关信息获取与写作，先参考 `feishu-use-skills-map` 并按其指引调用对应技能
- 需要访问飞书文档/群消息/会议/任务时，必须先 `get_skills` 获取对应飞书技能说明
- 内部知识库查询必须在外部检索之前或同步进行，确保内部视角不遗漏

### 工具/Skill 明确清单（执行前先 get_skills）
- 必须先参考 `feishu-use-skills-map` 并调用对应技能
- 主要技能及用途：

| 工具 | 用途 | 适用场景 |
|------|------|---------|
| `web_search` | 广泛检索公开信息 | 公司新闻、财报数据、行业排名、融资动态 |
| `fetch` | 获取特定网页/文档全文 | 公司官网、招股书、年报、深度报道全文 |
| `aily-doc` | 检索飞书云文档 | 内部已有的公司研究、分析报告、会议纪要 |
| `knowledge_answer` | 检索内部知识库 | 公司内部对该目标公司的分析、合作记录、竞品档案 |
| `file` | 将素材写入 draft.md（append 模式） | 每个数据步骤完成后 |

### 证据与时间约束（与主/写作agent一致）
- 任何数据/结论必须来自可追溯来源；不确定则明确标注"需补充资料"
- 若用户指定时间范围，必须严格筛选在时间窗内的资料，过期内容不得使用
- 不可编造事实、链接或来源；引用必须有明确出处
- 财务数据必须标注来源和对应年份/季度
- 多来源数据出现冲突时，优先采信公司官方披露，其次采信权威第三方，并在报告中注明差异

### 可读性规则（强制）
- 并列项>=3：优先使用无序/有序列表或表格
- 单段>150字必须拆分（列表/小节/Callout）
- 连续两段无列表且信息密度高：插入列表或 Callout
- 表格中数字右对齐，文字左对齐
- 时间线类信息使用有序列表，按时间倒序排列

### 触发方式（Triggers）
- "公司调研"
- "公司画像"
- "公司研究"
- "帮我研究XX公司"
- "帮我调研一下XX公司"
- "做一份XX公司的调研报告"
- "XX公司怎么样？帮我全面分析一下"
- "我想了解XX公司的详细情况"

### 输入（Inputs）
- **必选**：公司名称（含别名、英文名、股票代码等常见标识）
- **可选**：
  - 时间范围（默认：近 1-3 年数据，最新动态优先）
  - 关注维度（默认全维度；可指定：基本面/业务/竞争/财务/风险）
  - 对标公司（指定用于竞争对比的公司）
  - 地理范围（全球/中国/特定区域）
  - 研究深度（概览/标准/深度，默认标准）

### 核心输出（Deliverables）
- **飞书云文档公司调研报告**：结构标准、数据详实、来源可追溯
- 报告涵盖：公司概况、业务产品、竞争位置、经营财务、风险关注、重要动态、综合评价

### 质量标准
- **真实性**：关键事实有来源，财务数据标注年份和出处
- **结论克制**：不做无依据判断，推测性内容需明确标注
- **时效性**：优先使用最新数据，旧数据标注年份
- **完整性**：六大调研维度（基本面、业务、竞争、经营、风险、动态）均需覆盖
- **一致性**：同一指标在不同章节中数据口径保持一致

## 3) Module A — 信息收集（主 Agent 执行）

> **核心原则**：多工具、多维度、多来源，确保覆盖全面；优先使用飞书工具，必要时使用外部检索作为补充。每个子模块完成后立即将结构化内容写入 `draft.md`，为后续写作 Agent 提供完整素材。

### A1. 公司基本面

**目标**：获取公司的基本档案信息，包括成立背景、核心团队、融资上市等关键信息。

#### web_search 查询策略

第一轮 — 基础信息：
```
web_search: "{公司} 成立时间 总部 创始人 融资历程"
web_search: "{公司} 公司简介 企业概况 百科"
```

第二轮 — 资本市场信息：
```
web_search: "{公司} 上市 股价 市值 股票代码"
web_search: "{公司} IPO 招股书 估值"
```

第三轮 — 团队与治理：
```
web_search: "{公司} CEO 管理团队 核心高管"
web_search: "{公司} 组织架构 员工规模 团队背景"
```

#### fetch 深度获取

```
fetch(url: "{公司官网}/about", prompt: "提取公司简介、成立时间、总部地址、企业使命愿景、核心团队信息")
fetch(url: "{招股书/年报链接}", prompt: "提取公司发展历程、股权结构、主要股东、融资历程")
```

## 5) Module C — 调用 Writer Agent 生成报告

> **核心原则**：主 Agent 完成所有数据收集与分析后，通过 `task` 工具将写作任务分发给 Writer Agent，由 Writer Agent 完成最终报告的结构化输出。

### 任务分发

主 Agent 调用 `task` 工具，传递以下 JSON 指令：

```json
{
  "description": "生成公司调研报告云文档",
  "prompt": "基于 /home/workspace/draft.md 中的公司调研素材，生成一份结构完整的飞书云文档公司调研报告。\n\n要求：\n1. 严格按照公司调研报告模板结构\n2. 调研摘要放在最前面，便于快速阅读\n3. 所有数据标注来源和年份\n4. 结论有数据支撑，避免主观臆断\n5. 报告篇幅：标准版 10-20 页\n6. 表格格式统一，数字右对齐\n7. 风险部分需标注严重程度\n\n报告模板见下方。",
  "subagent_type": "writer"
}
```

### Writer Agent 执行流程

Writer Agent 收到任务后，按以下顺序执行：

```
步骤 1: file_read → 读取 /home/workspace/draft.md，获取全部调研素材
步骤 2: outline_generator → 根据模板结构和 draft.md 内容生成结构化大纲
步骤 3: feishu_doc_create → 根据大纲和素材生成飞书云文档
步骤 4: end → 返回飞书文档链接，结束任务
```

### Writer Agent 写作规范

- **标题层级**：使用 H1-H3，不超过 H4
- **表格规范**：表头加粗，数字右对齐，金额统一单位（亿元/万元）
- **引用标注**：关键数据后用括号标注来源和年份，如"营收 500 亿元（公司年报，2024）"
- **Callout 使用**：核心结论使用 Callout 高亮
- **图表建议**：在文档中标注建议插入图表的位置和类型

## 9) 常见失败模式与修复

| 失败模式 | 原因分析 | 修复方法 |
|---------|---------|---------|
| 基本面信息不全 | 只搜了一轮，遗漏创始人/融资/上市等维度 | 严格按 A1 三轮搜索策略执行，每轮针对不同维度 |
| 业务结构模糊 | 未获取营收占比数据，只有定性描述 | 必须从财报或权威报告中提取定量数据，无数据则标注"需补充资料" |
| 竞品对比不公平 | 对比维度不统一，或只选弱竞品 | 使用统一表格维度，选择真正的主要竞争对手（行业前 5） |
| 财务数据单一来源 | 只引用一个来源，数据可能有误 | 至少 2 个独立来源交叉验证，差异超 10% 需注明 |
| 财务数据年份混乱 | 不同指标取自不同年份，口径不一致 | 统一到同一财年，标注每个数据的具体年份 |
| SWOT 流于形式 | 优劣势过于笼统，没有具体事实支撑 | 每条 SWOT 要点必须附带"依据：{具体来源}" |
| 风险评估遗漏 | 只列了市场和竞争风险，忽略监管/技术/运营风险 | 严格按 8 类风险逐一评估，无风险可标注"当前无显著风险" |
| 内部视角缺失 | 跳过了 A6 内部资料检索 | knowledge_answer + aily-doc 必须执行，即使结果为空也要记录 |
| 动态信息过时 | 只检索到半年前的新闻 | 搜索关键词加上最新年份，如"2024 2025"，fetch 新闻页获取最新 |
| 结论无依据 | 给出判断但没有数据支撑 | 每个结论必须附带"依据"字段，推测性内容标注"需进一步验证" |
| draft.md 结构混乱 | 各模块写入格式不统一 | 严格按本 SKILL 中每个子模块的"写入 draft.md"模板执行 |
| 报告篇幅失控 | 堆砌过多细节，核心结论被淹没 | 标准版控制在 10-20 页，摘要部分确保 1 页内可读完 |
| 表格格式不统一 | 不同章节的表格风格差异大 | Writer Agent 统一格式：表头加粗、数字右对齐、金额统一单位 |
| 公司主体混淆 | 母子公司、关联公司信息混在一起 | 任务分解第一步"确认公司主体范围"必须明确研究边界 |
